1️⃣ Overview

What the Program Does

This is a bare-metal ARM Cortex-M4 firmware for a pedestrian zebra crossing traffic light controller running on the STM32F446RE NUCLEO-64 development board. It operates without an operating system, directly manipulating hardware registers.

  • Idle State: Pedestrian RED LED is on. Seven-segment displays show "STOP". Waits for button press.
  • Warning Phase: YELLOW LED turns on for 1 second to warn traffic.
  • Crossing Phase: GREEN LED turns on (via PWM for brightness control). A 60-second countdown is displayed on two 7-segment digits, alongside "GO" on four word-display digits. A buzzer beeps intermittently. During the last 10 seconds, the green LED blinks in a 2-second ON / 2-second OFF pattern.
  • Button Reset: If the button is pressed during crossing, the timer resets to 60 seconds.
  • End Crossing: Displays "00" for 1 second, then shows YELLOW for 1 second, then returns to IDLE.
  • Day/Night Mode: An independent timer toggles day/night mode every 10 seconds. In DAY mode, the green LED runs at full PWM brightness (999/999). In NIGHT mode, it runs dim (120/999).

Lines 27–35 — Assembler Directives & Vector Table 
27PRESERVE8           ; Guarantee 8-byte stack alignment (ARM AAPCS requirement)
28THUMB               ; Use Thumb-2 instruction set (Cortex-M only supports Thumb)
30AREA RESET, DATA, READONLY   ; Declare a read-only data section named RESET
31EXPORT __Vectors    ; Make __Vectors visible to the linker
33__Vectors
34    DCD  0x20020000  ; Word 0: Initial Stack Pointer (top of 128KB SRAM)
35    DCD  Reset_Handler ; Word 1: Reset vector (address of first instruction)
InstructionExplanation
PRESERVE8Tells the linker this module maintains 8-byte stack alignment, as required by the ARM Architecture Procedure Call Standard (AAPCS). The Cortex-M hardware also requires this for proper exception handling.
THUMBSelects the Thumb instruction set. Cortex-M processors only execute Thumb (16/32-bit mixed) instructions — ARM (32-bit fixed) mode is not available.
DCD 0x20020000Defines a 32-bit constant (Define Constant Data). The Cortex-M processor reads the first word of the vector table as the initial value for the Stack Pointer (SP). 0x20020000 is the end of the STM32F446RE's 128 KB SRAM region (0x20000000 + 0x20000).
DCD Reset_HandlerThe second word is the Reset vector — the address jumped to after reset. The linker will resolve Reset_Handler to an absolute address with bit 0 set (Thumb bit).

Lines 143–162 — Reset_Handler: Clock Initialization 
143Reset_Handler
148    LDR     R0, =RCC_AHB1ENR       ; R0 = 0x40023830
149    LDR     R1, [R0]              ; R1 = current AHB1ENR value

151    ORR     R1, R1, #(1 << 1)     ; Set bit 1 → enable GPIOB clock
152    ORR     R1, R1, #(1 << 2)     ; Set bit 2 → enable GPIOC clock
153    STR     R1, [R0]              ; Write back to AHB1ENR
158    LDR     R0, =RCC_APB1ENR       ; R0 = 0x40023840
159    LDR     R1, [R0]              ; R1 = current APB1ENR value
160    ORR     R1, R1, #(1 << 0)     ; Set bit 0 → enable TIM2 clock
161    ORR     R1, R1, #(1 << 1)     ; Set bit 1 → enable TIM3 clock
162    STR     R1, [R0]              ; Write back to APB1ENR
Why read-modify-write? The ORR pattern (LDR → ORR → STR) preserves other bits in the register. If we just wrote the desired bits directly, we would clear all other peripheral clocks. This is the standard STM32 register manipulation pattern.

Registers affected: R0 = register address, R1 = register value. After this block, GPIOA/B/C and TIM2/TIM3 clocks are enabled.

Lines 230–279 — Timer Configuration (TIM2 & TIM3)

TIM2 — 1 ms Tick Timer

230    LDR     R0, =TIM2_PSC
231    LDR     R1, =15999            ; Prescaler: 16 MHz / (15999+1) = 1 kHz
232    STR     R1, [R0]
234    LDR     R0, =TIM2_ARR
235    LDR     R1, =0xFFFFFFFF       ; Max auto-reload (TIM2 is 32-bit)
236    STR     R1, [R0]
238    LDR     R0, =TIM2_CNT
239    MOV     R1, #0              ; Reset counter to 0
240    STR     R1, [R0]
242    LDR     R0, =TIM2_EGR
243    MOV     R1, #1              ; UG bit → force update event
244    STR     R1, [R0]
246    LDR     R0, =TIM2_CR1
247    MOV     R1, #1              ; CEN bit → enable counter
248    STR     R1, [R0]
Timer Math: The STM32F446RE's default internal oscillator (HSI) runs at 16 MHz. With prescaler = 15999, the timer increments at 16,000,000 / (15,999 + 1) = 1000 Hz = 1 ms per tick. TIM2 is a 32-bit timer, so it counts up to ~4.29 billion ms (≈49.7 days) before wrapping.

TIM3 — PWM on PB4 (Green LED)

253    LDR     R0, =TIM3_PSC
254    MOV     R1, #15             ; Prescaler: 16 MHz / (15+1) = 1 MHz
257    LDR     R0, =TIM3_ARR
258    LDR     R1, =999            ; Period: 1 MHz / (999+1) = 1 kHz PWM
261    LDR     R0, =TIM3_CCR1
262    MOV     R1, #0              ; Initial duty cycle = 0 (LED off)
265    LDR     R0, =TIM3_CCMR1
266    MOV     R1, #0x68           ; OC1M=110 (PWM Mode 1) + OC1PE=1 (preload)
269    LDR     R0, =TIM3_CCER
270    MOV     R1, #0x01           ; CC1E=1 → enable CH1 output
277    LDR     R0, =TIM3_CR1
278    MOV     R1, #0x81           ; ARPE=1 + CEN=1 → auto-reload preload + enable
CCMR1 = 0x68: Bits [6:4] = 110 → PWM Mode 1 (output high when CNT < CCR1). Bit 3 (OC1PE) = 1 → preload enable (new CCR1 values take effect at next update event, preventing glitches).
CR1 = 0x81: Bit 7 (ARPE) = 1 → auto-reload preload. Bit 0 (CEN) = 1 → counter enabled. PWM frequency = 1 MHz / 1000 = 1 kHz.

LTORG directive: Forces the assembler to emit a literal pool at this location. Because LDR Rn, =constant generates a PC-relative load from a literal pool, the pool must be within ±4 KB of the instruction. Placing LTORG after branches ensures the pool doesn't get executed as code.

Lines 539–606 — UPDATE_TIMERS Function

This is the core timing engine. Called frequently from every busy-wait loop.

539UPDATE_TIMERS
540    PUSH    {R0-R7, LR}        ; Save 9 registers to stack
542    LDR     R0, =TIM2_CNT
543    LDR     R1, [R0]             ; R1 = current TIM2 count
545    LDR     R0, =LAST_TIM2_COUNT
546    LDR     R2, [R0]             ; R2 = previous TIM2 count
547    STR     R1, [R0]             ; Update stored count
549    SUBS    R3, R1, R2           ; R3 = delta (ms elapsed since last call)

The delta calculation R3 = current - previous gives milliseconds elapsed. Since TIM2 is 32-bit, unsigned subtraction handles wraparound correctly.

Day/Night Timer (Lines 554–573)

554    LDR     R0, =DAYNIGHT_ACC
555    LDR     R4, [R0]             ; R4 = accumulated ms
556    ADDS    R4, R4, R3           ; Add delta
558DAYNIGHT_CHECK_LOOP
559    LDR     R5, =DAYNIGHT_PERIOD_MS ; 10000
560    CMP     R4, R5
561    BLO     DAYNIGHT_STORE     ; If acc < 10000, done
563    SUBS    R4, R4, R5           ; acc -= 10000
565    LDR     R6, =MODE_FLAG
566    LDR     R7, [R6]
567    EOR     R7, R7, #1           ; Toggle: 0↔1 (DAY↔NIGHT)
568    STR     R7, [R6]
570    B       DAYNIGHT_CHECK_LOOP ; Check again (handles lag)
EOR toggle trick: EOR R7, R7, #1 (exclusive OR with 1) flips bit 0. If MODE_FLAG was 0 (DAY), it becomes 1 (NIGHT) and vice versa. This is a classic single-bit toggle.

The loop handles edge cases where the delta spans multiple periods (e.g., if UPDATE_TIMERS wasn't called for 20 seconds, it toggles twice).

Common-Cathode vs Common-Anode

The countdown display uses common-cathode (CC): cathode is shared at ground, segments turn ON by setting HIGH. The word display uses common-anode (CA): anode is shared at VCC, segments turn ON by setting LOW (inverted logic).

  • SET_SEGMENTS_CC: Uses ORRNE to set segment pins HIGH when the corresponding bit is 1.
  • SET_SEGMENTS_CA: First sets all segments HIGH (OFF), then uses BICNE to clear pins LOW (ON) for active segments. This is the inverted logic for common-anode.

4️⃣ ARM Concepts Used

Stack Operations

The program uses PUSH and POP for function prologue/epilogue. The stack grows downward from 0x20020000.

PUSH    {R0, R1, LR}   ; Save caller-saved regs + return address
; ... function body ...
POP     {R0, R1, PC}   ; Restore regs, LR value → PC = return

The POP {…, PC} pattern is the standard Thumb-2 function return: the saved LR value is loaded directly into PC, causing a branch to the return address. This is equivalent to POP {…, LR} followed by BX LR but saves one instruction.

Maximum stack depth: UPDATE_TIMERS pushes 9 words (36 bytes), and it calls no deep sub-functions, so the deepest nesting is approximately IDLE → DISPLAY_STOP_ONCE → SHOW_WORD_DIG1 → ALL_DIGITS_OFF → SET_SEGMENTS_CA, consuming roughly 4–5 frames × ~3–9 words = ~80–180 bytes of stack. Extremely modest.

Link Register (LR) & Program Counter (PC)

LR (R14): When BL label is executed, the processor stores the return address (next instruction) in LR, then branches to label. Nested calls overwrite LR, which is why every function pushes LR on entry.

PC (R15): The program counter. Writing to PC causes a branch. POP {…, PC} pops the saved LR into PC, effectively returning from a function.

Thumb bit: On Cortex-M, bit 0 of any branch target must be set to 1 (indicating Thumb mode). The BL instruction and the linker handle this automatically. If you manually loaded an address without the Thumb bit, a Usage Fault would occur.

Conditional Execution

ARM (and Thumb-2) supports conditional execution suffixes. In this code:

TST     R2, #(1 << 0)   ; Sets Z flag: Z=1 if bit 0 is 0
ORRNE   R1, R1, #PC0_SEGA ; Execute ORR only if Z=0 (bit was set)

This avoids a branch instruction, making the code smaller and faster. The NE suffix means "Not Equal" (Z flag = 0). Similarly, BICNE conditionally clears bits.

IT blocks: In Thumb-2, conditional instructions outside IT blocks are limited. The assembler automatically generates IT (If-Then) blocks for these conditional instructions. For example, ORRNE is assembled as IT NE followed by ORR.

Load/Store Architecture & Memory-Mapped I/O

ARM is a load/store architecture: data must be loaded from memory into registers before operations, then stored back. You cannot directly operate on memory contents.

STM32 peripherals are memory-mapped: GPIO, timers, clocks all appear as addresses in the 0x40000000–0x5FFFFFFF range. Writing to 0x40020414 (GPIOB_ODR) directly changes the voltage on Port B output pins.

The standard pattern throughout the code:

LDR  R0, =PERIPHERAL_REG  ; 1. Load register address
LDR  R1, [R0]              ; 2. Read current value
ORR  R1, R1, #BIT_MASK    ; 3. Set bits (or BIC to clear)
STR  R1, [R0]              ; 4. Write back
Memory Alignment

The ALIGN directive at line 1168 ensures the next data is word-aligned (4-byte boundary). This is important because:

  • LDR/STR require word-aligned addresses for 32-bit accesses
  • The DCD variables in MYVARS are 4 bytes each and naturally align after the DCB table in MYDATA
  • Unaligned access on Cortex-M4 is supported but slower; the ALIGN is good practice

🔟 Important Observations

✅ Clever Design Choices
  • Bit-9 buzzer toggle: Using bit 9 of the free-running TIM2 counter for buzzer timing is elegant — it creates a ~1 Hz tone without any additional counter variable or timer channel. Zero software overhead.
  • EOR toggle: EOR R7, R7, #1 for the day/night toggle is the optimal single-bit flip — one instruction, no branches.
  • Accumulator-drain loops: The while (acc >= period) { acc -= period; action(); } pattern correctly handles timing lag — if the main loop was slow for one iteration, multiple seconds/toggles are processed in the next call. No ticks are lost.
  • Counting semaphore: SECOND_FLAG acts as a counting semaphore (producer-consumer pattern). UPDATE_TIMERS produces (increments) and CONSUME_SECOND_FLAG consumes (decrements). This decouples the timer from the consumer.
  • Conditional execution: Using ORRNE/BICNE for segment mapping avoids 7 branch instructions per digit, saving both code size and execution time.
  • POP {…, PC}: The return-via-PC-pop pattern is a standard ARM optimization that saves one instruction per function return.
⚠️ Potential Issues & Inefficiencies
  • No debounce during crossing: The button reset during crossing (line 473–479) has NO debounce delay. A single button press could be detected across multiple loop iterations (button held for >1 loop), repeatedly resetting the timer. However, since the reset just reloads 60, repeated resets have the same effect, so this is benign in practice.
  • Hard-coded blink pattern: The blink logic (lines 437–458) uses a cascade of CMP/BGT instructions with hard-coded thresholds. A more maintainable approach would use modular arithmetic: blink_on = ((countdown - 1) / 2) % 2 == 1. However, the hard-coded version is arguably clearer for readers.
  • Division by repeated subtraction: CALC_TENS_LOOP divides by 10 using repeated subtraction. For a max value of 60, this means at most 6 iterations — negligible, but a hardware divide instruction (UDIV, available on Cortex-M4) would be faster for larger values.
  • Busy-wait delays: SHORT_DELAY and DELAY_SHORT use busy-wait loops. These waste CPU cycles and block other processing. In a production system, these would use timer interrupts or the SysTick timer. Acceptable for a prototype.
  • No WFI (Wait For Interrupt): The polling loops burn CPU at 100% constantly. Using WFI between timer checks would dramatically reduce power consumption — important for a battery-powered crossing controller.
  • Display flicker: Each digit is shown for only ~0.56 ms (MUX_DELAY = 3000). With 6 digits to multiplex per pass, the full cycle takes ~3.4 ms (~294 Hz refresh). This is adequate but could flicker on fast cameras. Increasing MUX_DELAY would reduce flicker but slow the main loop.
🐛 Potential Bugs
  • SECOND_FLAG race condition (minor): UPDATE_TIMERS and CONSUME_SECOND_FLAG both read-modify-write SECOND_FLAG. Since there are no interrupts and the code is single-threaded polled, this is not actually a bug. However, if interrupts were ever added, this would become a critical race condition requiring atomic operations or critical sections.
  • No minimum vector table: The vector table has only 2 entries (SP + Reset). If any fault occurs (HardFault, NMI, etc.), the processor will read garbage from unmapped memory as the handler address, likely entering an infinite fault loop. In production, at least a HardFault handler should be defined.
  • TIM2 CNT wrap-around: Subtraction R3 = R1 - R2 works correctly for 32-bit unsigned wrap-around (e.g., 0x00000005 - 0xFFFFFFF0 = 0x00000015 = 21). However, if UPDATE_TIMERS is not called for >49.7 days (2³² ms), the delta would be wrong. Not a realistic concern.
  • Green PWM called repeatedly: In the countdown loop, GREEN_PWM_CURRENT_MODE or GREEN_PWM_OFF is called every iteration, writing to TIM3_CCR1 each time. With preload enabled (OC1PE=1), this is safe but unnecessary — a flag could skip redundant writes.
🔒 Security Concerns
  • No memory protection: The code runs in privileged mode with full access to all peripherals. An errant pointer could write to any peripheral register or overwrite SRAM. The Cortex-M4's MPU (Memory Protection Unit) is not configured.
  • No watchdog timer: If the software hangs (e.g., a tight loop bug), there is no watchdog to reset the system. A production crossing controller should enable IWDG (Independent Watchdog).
  • No stack overflow protection: The stack grows toward the variables in SRAM. With only ~44 bytes of stack used and 128 KB of SRAM, overflow is extremely unlikely, but there is no guard region or canary.
  • Button can extend crossing indefinitely: The timer-reset feature (line 478) means a malicious user could hold the button to keep the crossing green forever, blocking traffic. A production system should have a maximum total crossing time.
💡 Interesting Low-Level Tricks
  1. Dual-type 7-segment driving: The same physical segment GPIO pins drive both common-cathode (countdown) and common-anode (word) displays. The code switches between SET_SEGMENTS_CC and SET_SEGMENTS_CA functions, which invert the driving logic (ORR vs BIC). This is uncommon and saves GPIO pins.
  2. LTORG placement: The code carefully places LTORG directives after unconditional branches (B) and before new labels. This ensures literal pools are: (a) reachable from nearby LDR Rn, =const instructions, and (b) never executed as instructions.
  3. Timer as RNG-like source: Using TIM2_CNT bit 9 for buzzer timing means the buzzer phase depends on when the loop happens to read the counter. This creates a pseudo-random feeling to the beep timing, though it's actually deterministic.
  4. Zero-overhead mode change: The day/night mode changes are computed during every UPDATE_TIMERS call but only applied when the green LED is actually being driven. During IDLE, mode changes happen silently in the background. This is an elegant separation of concerns.
